# スキーマ駆動開発 vs コードファースト開発：比較と検証計画

## 🔍 比較の目的

本プロジェクトでは、**同一仕様のCRUD API**（書籍管理）を3つの手法で実装し、設計・開発・保守の観点からそれぞれの開発アプローチを比較・検証する。

- OpenAPI（スキーマ駆動）
- gRPC（スキーマ駆動）
- コードファースト（手動定義）

---

## 🧪 比較の形式

### ✅ 実施内容

| 手法         | 実装内容                            | 使用技術例                       |
|--------------|-------------------------------------|----------------------------------|
| OpenAPI      | OpenAPI仕様からAPIを生成・実装     | Swagger、Express、YAML          |
| gRPC         | `.proto`からサーバ・クライアント生成| Protobuf、gRPC、Node.js         |
| コードファースト | モデル／ルートを手動で定義           | Express、JavaScript             |

全て、同一機能（例：書籍の登録・取得・更新・削除）を実装。

---

## 🎯 比較の評価基準

| 評価軸             | 説明 |
|----------------------|------|
| **初期実装のしやすさ** | 設計〜初期開発までの流れがスムーズか（時間、ステップ数） |
| **保守性**            | 仕様変更に対してコードの変更が容易か／追従性があるか |
| **拡張性**            | 新しい機能やリソースを追加しやすいか |
| **ドキュメント性**    | API仕様が自動生成・共有しやすいか（Swagger UI など） |
| **テストのしやすさ**  | 型安全性やモックの容易さによるテスト支援の有無 |
| **学習コスト／導入難易度** | 初学者／チームでの導入しやすさ |
| **ツールサポート**    | Swagger / Postman / gRPC UI などの対応状況 |
| **通信効率**          | 特にREST vs gRPCの通信効率・速度の違い |
